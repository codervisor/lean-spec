# `leanspec crystallize` — Codebase Scanner

## Overview

AI coding agents work best when they understand a codebase's conventions,
patterns, and decisions. Today, teams write `AGENTS.md` and `.claude/skills/`
files manually — a process that is slow, inconsistent, and drifts as the
codebase evolves.

`leanspec crystallize` scans the local codebase, analyzes git history, and
reads the project's LeanSpec specs to automatically generate L1 rules
(`AGENTS.md` / `CLAUDE.md`) and L2 procedural knowledge (`.claude/skills/`).

This is the first concrete deliverable toward the Codervisor platform vision
(spec 380) that requires no external infrastructure.

Done when: `leanspec crystallize` runs against the lean-spec repo itself and
produces a useful `AGENTS.md` and at least one skill file.

## Design

### L1 and L2 rule levels

**L1 — Project rules** → written to `AGENTS.md` or `CLAUDE.md`
- Architecture invariants ("All adapters must implement the compliance test suite")
- Naming conventions ("Commands are in `commands/{name}.rs`")
- Error handling patterns ("Use `AdapterError` for all adapter failures")
- Forbidden patterns ("Never import `SpecLoader` from outside `adapters/markdown/`")

**L2 — Procedural knowledge** → written to `.claude/skills/{topic}.md`
- How to add a new command
- How to add a new adapter
- How to run the test suite
- How to release a new version

### Scanners

`crystallize` runs a set of scanners in parallel. Each scanner is independent
and returns a list of `Signal` items. Signals are weighted and deduplicated
before being passed to the rule generator.

**1. Architecture pattern scanner**

Scans source files for recurring structural patterns:
- Module layout: which modules are in which crates
- Trait/impl relationships: which types implement which traits
- Error type usage: `thiserror` vs `anyhow` vs custom
- Import patterns: what's imported from where (detects coupling violations)

Output signals:
- "All adapter implementations are in `rust/leanspec-core/src/adapters/`"
- "Commands are in `rust/leanspec-cli/src/commands/`"
- "Tests use the `TestContext` helper from `tests/common.rs`"

**2. Naming convention scanner**

Scans identifiers for consistent naming patterns:
- File naming (snake_case modules, hyphenated specs)
- Function naming (verb-first imperative for commands)
- Type naming (PascalCase, `Adapter` suffix for adapter types)

Output signals:
- "Rust: snake_case modules, PascalCase types"
- "CLI command functions are named `run(params: XParams)`"

**3. Git history scanner**

Reads `git log --follow -p` for recurring commit patterns:
- Files changed together frequently → architectural coupling
- Files never changed → stable contracts
- PR merge patterns → review requirements
- Recurring commit message prefixes → types in use

Output signals:
- "spec_ops/ and adapters/markdown/ are always changed together"
- "Cargo.toml changes always accompanied by Cargo.lock"
- "Commit message types in use: feat, fix, refactor, test, docs, chore"

**4. Spec intent scanner**

Reads all LeanSpec specs (via `adapter.list()`) and extracts:
- Architectural decisions from ADR specs
- "Done when" criteria from feature specs
- Constraints from Notes sections

Output signals:
- "Never use `DynamicStatus` shims (spec 394)"
- "HTTP handlers use fetch-transform-push for body manipulation (spec 387)"
- "All adapters must pass the compliance test suite (spec 385)"

**5. Test pattern scanner**

Scans test files for testing conventions:
- Test helper usage (`TestContext`, `mockito`, fixture files)
- Test naming patterns
- Integration test guards (`#[cfg(feature = "...")]`)

Output signals:
- "E2E tests use `TestContext` from `tests/common.rs`"
- "Integration tests require feature flags, not `#[ignore]`"

### Rule generation

After all scanners run, signals are grouped by theme and passed to a rule
generator. The generator uses string-matching heuristics (no LLM required for
the initial version — LLM integration is a future spec) to produce rule text.

Rules are categorized:
- `#[arch]` — architecture rules (what belongs where)
- `#[naming]` — naming conventions
- `#[testing]` — testing requirements
- `#[forbidden]` — anti-patterns to avoid
- `#[workflow]` — procedural L2 rules

### Output format

**`AGENTS.md` / `CLAUDE.md`** (L1 rules):

```markdown
# Project Rules (generated by `leanspec crystallize`)
<!-- Generated: 2026-05-14. Re-run `leanspec crystallize` to update. -->

## Architecture

- All adapters implement the `Adapter` trait in `rust/leanspec-core/src/adapters/`
- HTTP handlers use the fetch-transform-push pattern for body manipulation
- Never import `SpecLoader` outside `adapters/markdown/`

## Testing

- E2E tests use `TestContext` from `rust/leanspec-cli/tests/common.rs`
- Integration tests requiring network are guarded by feature flags, not `#[ignore]`
- All new adapters must pass the compliance suite in `spec 385`

## Conventions

- Commit message types: feat, fix, refactor, test, docs, chore, ci, perf
- Command functions are named `run(params: XParams) -> Result<(), Box<dyn Error>>`
```

**`.claude/skills/{topic}.md`** (L2 procedural):

```markdown
# How to Add a New Adapter

1. Create `rust/leanspec-core/src/adapters/{name}.rs`
2. Implement the `Adapter` trait (all 7 methods)
3. Add `resolve_schema()` to fetch live field options
4. Register in `AdapterRegistry::create()` match arm
5. Add unit tests with canned JSON fixtures
6. Run the compliance test suite (spec 385)
7. Update `leanspec init --adapter {name}` in spec 388's commands/init.rs
```

### Modes

```
leanspec crystallize                 # generate + write to AGENTS.md + skills/
leanspec crystallize --dry-run       # print output, write nothing
leanspec crystallize --output stdout # print to stdout only
leanspec crystallize --update        # update existing files (merge, not replace)
```

`--update` mode reads existing `AGENTS.md`, keeps manually-added rules
(marked with `<!-- manual -->`), and replaces only the generated sections.

## Notes

### No LLM dependency in v1

The initial implementation uses heuristic rule generation — pattern matching
and template filling, not LLM inference. This keeps the command fast (< 10s),
offline-capable, and deterministic. LLM-enhanced crystallization (using Claude
to synthesize richer rules) is a future enhancement.

### Relation to Codervisor platform

Spec 380 outlines a platform with knowledge graph, multi-repo scope, and
Synodic integration. This spec delivers only the **local codebase scanner** —
the first tier. It is deliberately scoped to be useful standalone without any
platform dependency.

### Update frequency

Teams should run `leanspec crystallize` when:
- The architecture changes significantly
- New patterns are established
- A major refactor completes

Automating it as a post-merge CI step is a future enhancement.
